# Архитектура проекта

Цель: дать переиспользуемый baseline для архитектуры проекта без привязки к конкретному продукту.

База: DDD (bounded contexts) + Clean Architecture (зависимости “снаружи внутрь”) + явный инвентарь деплоя + единый каркас директорий.

Устаревшие и архивные каталоги не считаются активной структурой проекта. Новая реализация создаётся в целевой структуре, а не восстанавливается механическим переносом старого кода.

## Что проект должен зафиксировать локально

Этот пакет не задаёт обязательные ограничения конкретного проекта. В локальной документации проекта должны быть явно описаны:

- поддерживаемый оркестратор или набор оркестраторов;
- схема аутентификации и доступа;
- список repository providers и правила интеграции с ними;
- модель запуска процессов
  (например, `webhook/event-driven`, `queue-driven`, `workflow-driven`);
- канонический state backend и правила хранения гибких данных;
- naming conventions для env, secrets, images и deploy inventory.

## Рекомендуемая структура репозитория

Целевой верхний уровень:

- `services.yaml` или его согласованная замена — инвентарь деплоя и окружений.
- `services/` — сервисы по зонам (`internal|external|staff|jobs|dev`) или по другой заранее согласованной схеме проекта.
- `libs/` — переиспользуемый код (`go|ts|vue|...`), создаётся только при наличии реального переиспользования.
- `proto/` — gRPC контракты, если проект использует gRPC.
- `deploy/` — инфраструктурные манифесты и overlays.
- `bootstrap/` — bootstrap-скрипты и первичная настройка окружений.
- `docs/` — проектная документация и решения.
- `tools/` — утилиты и генерация, создаются при необходимости.

## Изменение состава deployable-сервисов

Если в проекте добавляется, удаляется или перестраивается deployable-сервис:

- синхронно обновлять сборку и публикацию образов;
- обновлять deploy inventory, runtime vars/secrets и манифесты;
- обновлять bootstrap и документацию;
- фиксировать порядок выкладки и ожидание зависимостей для production-пути.

Проектные naming conventions задаются локально. Например, проект может использовать шаблон вида
`<PROJECT>_<SERVICE>_IMAGE`.

## Бюджеты исполнения и лимиты нагрузки

Каждый deployable-сервис должен иметь явный бюджет ресурсов и ограничители нагрузки. Это нужно,
чтобы отказ одного сервиса, клиента или внешней зависимости не превращался в лавину соединений,
запросов, потоков исполнения или подключений к хранилищам.

Общие правила:

- лимиты входящих HTTP/gRPC/queue операций задаются через конфигурацию, а не числовыми литералами в коде;
- сервис должен иметь ограниченный буфер входящих операций и понятную реакцию на переполнение:
  отказ, backpressure или деградацию по заранее описанному сценарию;
- метрики активных операций, занятости буфера, отказов из-за переполнения и длительности ожидания
  должны быть пригодны для автомасштабирования и оповещений;
- пулы БД, кэша, внешних API и очередей должны иметь явные максимумы, время жизни соединений,
  idle-настройки и retry/backoff с jitter;
- проект должен учитывать суммарную ёмкость внешней зависимости: лимит на одну реплику нельзя выбирать
  без учёта максимального числа реплик;
- не допускаются неограниченные фоновые задачи, worker pools, goroutines, promises или timers;
- все лимиты параллельности, размеры пулов, размеры очередей, таймауты и retry/backoff параметры должны быть
  частью типизированной конфигурации с валидацией на старте.

Если проект использует HPA или другой механизм автомасштабирования, масштабирование должно опираться
не только на CPU/RAM, но и на сигналы насыщения сервиса: активные операции, глубину очередей и буферов,
давление соединений, отказы из-за лимитов и задержку.

## Зоны сервисов: пример

Если проект использует зонирование по типам сервисов, оно может выглядеть так:

### `services/internal/`
- Доменные правила и каноническое состояние.
- Работа с БД, оркестратором и provider APIs через интерфейсы.

### `services/external/`
- Публичные webhook/API точки входа.
- Валидация подписи, authn/authz, rate limiting, аудит.

### `services/staff/`
- Внутренние операторские интерфейсы.
- Доступ через выбранную схему SSO/OAuth и проектную матрицу прав.

### `services/jobs/`
- Async/фоновые процессы: reconciliation, ретраи, cleanups, индексация.

### `services/dev/`
- Dev-only инструменты.
- Не деплоятся в production.

Проект может использовать и другую модель разбиения, если она описана явно и не размывает доменные границы.

## Границы ответственности

Правила:

- Один сервис = один bounded context и одна причина для изменения.
- Домен не зависит от transport/DB SDK напрямую.
- Интеграции с оркестратором, repository providers и внешними API изолированы в адаптерах.
- Shared DB без владельца контекста запрещён; таблицы и данные имеют явного владельца.

## Схема взаимодействия

Конкретная схема взаимодействия определяется архитектурной документацией проекта. Этот гайд фиксирует только общий baseline:

- внешние точки входа валидируют вход и маршрутизируют запросы;
- доменные сервисы владеют бизнес-логикой и каноническим состоянием;
- фоновые процессы выполняются идемпотентно;
- операторские интерфейсы работают через API, а не напрямую с БД;
- интеграции с внешними системами идут через интерфейсы и адаптеры.
